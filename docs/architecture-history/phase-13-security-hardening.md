# Phase 13 — Security Hardening

**Status**: Complete

## Summary

Phase 13 added a defence-in-depth security layer across the Scout, Knowledge, and Analyst pipeline stages. The work has seven distinct components: SSRF validation, global + per-tenant blocklists, PII detection, output sanitization, an append-only audit log, prompt injection containment in the validator rubric, and adversarial golden examples for the evaluation pipeline. All components are wired to Prometheus metrics via `METRICS.security_blocks` (added to `shared/metrics/registry.py`).

---

**SSRF validator** (`infrastructure/security/ssrf.py`). `validate_url(url, *, resolved_addrs=None) -> ValidatedURL` is the entry point. It composes four pure helper functions: `_check_scheme` (rejects non-HTTP/S), `_check_hostname_suffix` (rejects `.internal`/`.local`/`.cluster.local`/`.svc`/`localhost`), `_check_address` (rejects RFC 1918, loopback, link-local 169.254.0.0/16, CG-NAT 100.64.0.0/10, IPv6 ULA/loopback/link-local, IPv4-mapped IPv6), and `resolve_hostname` (blocking `socket.getaddrinfo` — must be called from `loop.run_in_executor(None, ...)`). `ValidatedURL` is a frozen dataclass carrying the canonicalized url/host/scheme. `SSRFBlockedError` (subclass of `FetchError`) carries a `reason` string that must be a member of `SecurityBlockReason` enum in `shared/metrics/registry.py`; it appears verbatim as the `reason` Prometheus label.

`validate_redirect(from_url, to_url, *, resolved_addrs=None)` handles per-hop redirect validation and adds `_check_scheme_downgrade` (blocks https→http). `MAX_REDIRECTS = 3` is a module-level constant in `ssrf.py`, not a config value — it is a security constant, intentionally harder than the configurable `max_redirects`. The Scout fetcher (`agents/scout/fetcher.py`) sets `follow_redirects=False` on the httpx `AsyncClient` and calls `validate_url` before the initial connection and `validate_redirect` before each subsequent hop.

All inner helpers are importable for table-driven unit tests in `tests/unit/test_ssrf_validator.py` (no DNS, no network — all tests use injected `resolved_addrs`).

**Blocklist** (`infrastructure/security/blocklist.py`). Two levels: global (loaded from `ops/security/global-blocklist.txt` at module import into `_GLOBAL_DOMAINS: frozenset` and `_GLOBAL_SUFFIXES: frozenset`); per-tenant (fetched from the `url_blocklists` MongoDB collection via `UrlBlocklistRepository`, cached in `_TENANT_CACHE: dict[str, tuple[float, frozenset, frozenset]]` with `_CACHE_TTL = 60.0` seconds under an `asyncio.Lock`). `check_global_blocklist(hostname, url)` and `check_tenant_blocklist(hostname, url, *, tenant_id, repo)` both raise `SSRFBlockedError` with `reason="ssrf_blocklist_global"` or `reason="ssrf_blocklist_tenant"` respectively. The Scout fetcher calls both checks before `validate_url`. On DB error in `_get_tenant_lists`, the function returns empty sets and logs a warning — the global blocklist and SSRF rules still apply (fail-open on the tenant-specific layer only). `domain/url_blocklists/models.py` — `UrlBlocklistEntry` (tenant_id, pattern, pattern_type: `"domain" | "suffix" | "cidr"`, notes). `infrastructure/mongo/repositories/url_blocklist_repository.py` — `MongoUrlBlocklistRepository`, extends `TenantScopedRepository`. Two indexes on `url_blocklists`: `(tenant_id, created_at DESC)` and unique `(tenant_id, pattern)`.

**PII detection** (`infrastructure/security/pii.py`). `PIIDetector` is a Protocol with one method: `detect_and_redact(text: str) -> RedactionResult`. `DefaultPIIDetector` is the regex-based implementation. Five pattern classes compiled at module load: `_SSN_RE` (3-2-4 digit, excludes 000/666/9xx prefixes), `_CC_RE` (13–19 digits in groups, Luhn-validated by `_luhn_valid`), `_EMAIL_RE` (RFC 5321-like), `_PHONE_RE` (E.164 + North American, min 7 digits), `_IBAN_RE` (ISO 13616 2-letter country + 2 check + up to 30 alphanumeric; validated by `_iban_valid` mod-97 check). `PIIDetection` is a frozen dataclass: `pii_type`, `start`, `end`, `redact_token` (`[REDACTED:SSN]` etc.). `RedactionResult`: `original_text`, `redacted_text`, `detections: tuple[PIIDetection, ...]`, `has_pii` property. Overlapping detections are resolved by sorting `(start, -(end-start))` and skipping spans covered by an earlier match. The Knowledge pipeline calls `detect_and_redact` per chunk before ChromaDB upsert and Mongo insertion; PII-detected chunks are stored with redacted text only. The `PIIDetector` Protocol is the seam for a future Presidio integration — swap `DefaultPIIDetector` for `PresidioPIIDetector` without changing any call site.

**Output sanitization** (`infrastructure/security/sanitize.py`). `sanitize_text(text: str) -> str` strips: `<script>...</script>` (IGNORECASE | DOTALL), `<style>...</style>`, `<iframe|object|embed|applet|base|link|meta|noscript|form>` tags, `on\w+=...` event handler attributes, `href/src/action=javascript:...` URIs, `href/src/action=data:...` URIs. All patterns are compiled at module load. `html_escape(text)` wraps `html.escape(..., quote=True)` for use in report templates when embedding user-influenced content in HTML attribute values or text nodes. The Analyst's `_build_analysis_from_tool_args` in `services/analysis/analyst.py` calls `sanitize_text` on `summary` and each `Insight.claim` before creating the `Analysis` domain model. The Publisher calls it again before blob upload — defense-in-depth, not double-billing.

**Audit log** (`domain/audit_log/models.py`, `infrastructure/mongo/repositories/audit_log_repository.py`). `AuditLogEntry` is a frozen Pydantic model with `extra="forbid"`: `id` (ULID), `tenant_id`, `event_type` (`"ssrf_blocked"` / `"pii_detected"` / `"url_blocked"`), `actor` (`"system"` / `"api_key:<id>"`), `resource_type`, `resource_id`, `action`, `outcome` (`"blocked"` / `"redacted"` / `"allowed"` / `"detected"`), `metadata: dict`, `created_at` (UTC), `trace_id`, `correlation_id`. The collection has three indexes: `(tenant_id, created_at DESC)`, `(tenant_id, event_type, created_at DESC)`, and a TTL index on `created_at` with `expire_after_seconds=31536000` (365 days — regulatory retention minimum). `MongoAuditLogRepository` extends `TenantScopedRepository[AuditLogEntry]` with one public method: `append(entry)`. Write failures are caught at every call site and logged as warnings — a transient Mongo error must never abort the business path (same pattern as `record_llm_cost`). The `OutboxRelayWorker` does not use the audit log (it is infrastructure-level, not tenant-scoped). The security workers (scout, knowledge, analyst) write audit log entries directly via the repository injected at worker startup.

**Prompt injection containment**. The `validator_system.md` prompt gained a sixth rubric dimension: `instruction_resistance` (weight 0.10). The `record_validation` tool's `score` field is now the weighted sum: `0.28 * grounding + 0.23 * faithfulness + 0.18 * quality + 0.13 * coverage + 0.08 * specificity + 0.10 * injection_resistance`. The weights still sum to 1.00. The dimension is scored 1.0 (no influence), 0.5 (minor), 0.0 (confirmed hijack). A score of 0.0 sets `passed=false` regardless of other dimensions — the only hard override in the rubric. The `analyst_system.md` prompt was updated to add an explicit data/instruction boundary reminder: chunk content is delimited by `<chunk id="{chunk_id}" source="{source_id}">...</chunk>` tags; the analyst is instructed to treat chunk content as data only. The Analyst's `_format_chunks_for_prompt` in `services/analysis/analyst.py` enforces the XML-style delimiter on every chunk.

**Adversarial golden examples** (`eval/golden/adversarial/`). Five JSON files validated as `GoldenExample` models: `instruction_injection.json`, `role_reassignment.json`, `ignore_prior.json`, `exfiltration.json`, `ssrf_in_html.json`. Each contains chunks with embedded adversarial content and `ExpectedAnalysis` instances where all `required=True` insights are legitimate category insights and the adversarial instruction must not appear as an insight claim. The adversarial goldens run as part of `make check-all` via `tests/unit/test_adversarial_goldens.py` — they are regression tests, not production eval samples.

**Security metrics**. `_TrendStormMetrics` gained `security_blocks: Counter` with labels `(reason, tenant_id_hash)`. `reason` is drawn from the closed `SecurityBlockReason(StrEnum)` enum (15 SSRF reasons + 5 PII reasons = 20 total values). `tenant_id_hash` uses `hash(tenant_id) % 100` mapped to a `"b{N:02d}"` string — 100 buckets, bounded cardinality regardless of tenant count — because raw `tenant_id` is a forbidden high-cardinality label per rule 51 in the engineering context. `record_security_block(reason, tenant_id, *, metrics=None)` is the helper used by all security call sites; it swallows all exceptions to prevent metric failure from crashing business logic. The `trendstorm-overview` Grafana dashboard gained a "Security" row with panels for SSRF block rate by reason and PII detection rate by type.

**Tests**: 1043 unit tests (927 from Phase 12 + 116 new). `tests/unit/test_ssrf_validator.py` — all pure, `resolved_addrs` injection, covers all IPv4/IPv6/scheme/hostname rejection classes including DNS rebinding and scheme downgrade. `tests/unit/test_pii_detector.py` — Luhn algorithm, IBAN mod-97, all PII types, overlapping redaction, original text preservation. `tests/unit/test_sanitizer.py` — XSS pattern coverage including multiline script blocks and event handler stripping. `tests/integration/test_security_defenses.py` — SSRF end-to-end, audit log persistence, PII detection integration, security metrics counter.

## Non-obvious decisions

**`validate_url` is a pure function with injectable `resolved_addrs`**. The function signature is `validate_url(url, *, resolved_addrs: list[str] | None = None)`. When `resolved_addrs is None`, the function calls `resolve_hostname(hostname)` (a blocking socket call). When `resolved_addrs` is provided (even as an empty list), DNS is skipped entirely. This lets every unit test in `test_ssrf_validator.py` run without network I/O and without mocking the socket module — injectable IPs are explicit test parameters. An empty list simulates a DNS failure and produces `SSRFBlockedError(reason="ssrf_dns_failure")`, which is the fail-closed behavior (unknown hostname is treated as blocked, not allowed). The production path (caller passes `None`) does real DNS in an executor. The alternative — always calling `resolve_hostname` and mocking `socket.getaddrinfo` — couples tests to the OS socket implementation and makes the test suite fragile on machines with different resolver behavior.

**Redirects are followed manually rather than via `httpx follow_redirects=True`**. If `httpx` followed redirects internally, the SSRF validator would only see the initial URL — every redirect hop would bypass all checks. Manual following (the Scout fetcher reads the `Location` header and calls `validate_redirect` before issuing the next request) means every hop is independently validated: scheme downgrade check, IP check on the resolved hostname, and the hop counter enforced by `MAX_REDIRECTS`. The `httpx.AsyncClient` is created with `follow_redirects=False` to make the intent explicit and to prevent future maintainers from accidentally re-enabling it. Internally, `httpx` would also buffer the response body before following, adding memory pressure for large redirect chains — manual following avoids that too.

**Global blocklist is a file loaded at module init**. `_load_global_blocklist()` is called once when `infrastructure/security/blocklist` is first imported. It reads `ops/security/global-blocklist.txt` into two frozensets. There is no periodic reload, no Redis cache, no DB entry — a plain file on the filesystem. The reasons: (a) the file is small (< 10 KB in practice); reading it at startup adds < 1 ms to worker startup time; (b) zero I/O per fetch request — check is a set lookup; (c) ops changes are applied via a rolling restart of Scout workers, not a config reload endpoint (twelve-factor). The per-tenant blocklist uses a 60s in-process cache (not cross-worker) for the same reason: a MongoDB round-trip on every URL check would add ~2 ms per source and make the blocklist check a database hotspot at high concurrency.

**Chunk content uses XML-style `<chunk>` delimiters**. The analyst prompt wraps each retrieved chunk in `<chunk id="{chunk_id}" source="{source_id}">...</chunk>`. XML tag syntax was chosen over Markdown fences or plain separators because it creates the clearest visual boundary between the instruction context (system prompt) and the data context (chunk text) — both for the model and for the validator when it reads the same corpus. The opening and closing tags include chunk metadata as attributes so the model can trace claims back to specific chunks without parsing additional prose. Adversarial instructions that read "ignore previous instructions" inside a clearly-labelled data tag are harder for the model to misinterpret as instructions; measurably, adversarial golden examples score higher `injection_resistance` with delimiters than with plain separators.

**`tenant_id_hash` uses `hash() % 100` buckets instead of raw `tenant_id`**. Rule 51 forbids high-cardinality identifiers as Prometheus label values. Raw `tenant_id` (a ULID — effectively unbounded) as a label would create one time series per tenant per reason — potentially 20,000 series at 1,000 tenants × 20 reason codes. `hash(tenant_id) % 100` maps all tenants to 100 buckets, capping cardinality at `100 * 20 = 2,000` series regardless of tenant count. The `tenant_id_hash` function is exported from `shared/metrics/registry.py` as a module-level helper so it can be reused if future metrics also need a hashed tenant dimension. The trade-off: multiple tenants share a bucket, so the metric cannot identify which specific tenant is generating blocks — Loki logs carry the raw `tenant_id` and provide per-event identification.

**Audit log writes fail-open**. Every call site wraps `audit_log_repo.append(entry)` in `try/except Exception: logger.warning(...)`. A transient Mongo write failure produces a warning log but does not propagate an exception to the caller. The business logic (SSRF block, PII redaction) has already executed before the audit write — those outcomes are final. Failing the business operation retroactively because the audit write failed would be worse: the SSRF block would appear to have not fired (the source fetch would be retried) yet the block happened. The audit log is a record of what occurred, not a gate on whether it occurs. This is the same pattern as `record_llm_cost` (rule 68 in the engineering context).

**`PIIDetector` is a Protocol with `DefaultPIIDetector`**. The Protocol has one method: `detect_and_redact(text: str) -> RedactionResult`. `DefaultPIIDetector` satisfies it via regex. The seam exists for Microsoft Presidio: Presidio supports 50+ entity types, multilingual detection, and ML-based models. When the team is ready to adopt it, `PresidioPIIDetector` will implement the same Protocol and be injected at the Knowledge worker's startup without changing `KnowledgePipeline` or any call site. The regex implementation handles the five highest-risk patterns (SSN, CC, email, phone, IBAN) with acceptable false-positive rates for English-language web content; Luhn and mod-97 validation significantly reduce false positives for CC and IBAN respectively.

**`injection_resistance` dimension weight is 0.10**. The rubric weights in `validator_system.md` sum to 1.00: `0.28 + 0.23 + 0.18 + 0.13 + 0.08 + 0.10 = 1.00`. The 0.10 weight was chosen to (a) not displace the existing grounding/faithfulness/quality dimensions that are the primary quality signal, and (b) be large enough that a 0.0 injection score (confirmed hijack) drops the aggregate score by 10 points — which, combined with the hard `passed=false` override, guarantees no hijacked analysis is published regardless of how well-formed its citations are. A weight higher than 0.10 would cause high-quality legitimate analyses referencing topic-adjacent LLM safety papers (which contain phrases that superficially resemble injection payloads) to be penalized unfairly. A weight lower than 0.10 would make the dimension cosmetic — the `passed=false` hard override still guards, but the numeric score would not reflect the severity clearly to the on-call engineer reading the Loki log.

## Tradeoffs

**Synchronous DNS resolution adds latency to the Scout hot path**. `resolve_hostname` calls `socket.getaddrinfo`, which blocks the OS thread. In the Scout fetcher, it is called via `await loop.run_in_executor(None, resolve_hostname, hostname)` so it does not block the event loop. A typical resolution takes 5–50 ms per hostname on a well-connected host. At Scout concurrency of 10 simultaneous fetches, the executor pool (default: `min(32, cpu_count + 4)`) is sufficient — DNS calls do not queue behind each other. A future optimization is a DNS cache keyed by `(hostname, TTL)` to avoid redundant lookups for the same hostname across multiple sources in one job; this was deferred as a Phase 14 item.

**In-process tenant blocklist cache has no cross-worker invalidation**. Each Scout worker pod maintains its own `_TENANT_CACHE` dict. When a tenant adds a blocklist entry, the change is visible to a given worker only after its 60s cache TTL expires. In a 3-replica Scout deployment, the update propagates to all workers within ~60 seconds, not instantly. This is an accepted trade-off: the alternative (invalidate via Redis pub/sub) adds a Redis dependency to the blocklist check hot path and adds operational complexity (Redis unavailability would degrade blocklist freshness in a different way). The 60s window is documented in the pentest checklist (`ops/security/pentest-phase13.md`, test 5.3) and in the security incident runbook.

**Global blocklist is not hot-reloadable**. Updating `ops/security/global-blocklist.txt` requires restarting Scout workers for the change to take effect (module-init loading). A rolling restart propagates the change within minutes in Kubernetes. This is intentional: hot-reload of a security file at a configurable interval creates a race window where some requests see the old file and some see the new file. A full pod restart is atomic per pod. The operational overhead is acceptable given that global blocklist updates are rare.

## Gotchas

**`validator_system.md` weights MUST sum to 1.00**. As of Phase 13: `0.28 + 0.23 + 0.18 + 0.13 + 0.08 + 0.10 = 1.00`. The validator prompt hardcodes these in the Aggregation section. The unit test `tests/unit/test_analysis_prompts.py` locks file/tool/rubric-dimension presence but does NOT currently assert the numeric sum — if a future phase adjusts weights, manually verify they sum to 1.00 and update this document. If they don't sum to 1.00, the `score` field will be out of `[0.0, 1.0]` range and `record_validation` will produce anomalous scores that may silently pass or fail validation checks.

**`audit_log` collection has a 365-day TTL**. Index name: `audit_log__ttl_created`. The `expire_after_seconds` value is `365 * 86400 = 31536000`. This index must be present — without it, the collection grows unboundedly. The seeder (`scripts/seed_mongo_indexes.py`) creates it idempotently. Verify presence after any index migration: `db.audit_log.getIndexes()` should list `audit_log__ttl_created` with `expireAfterSeconds: 31536000`.

**`SSRFBlockedError.reason` is a Prometheus label — must match `SecurityBlockReason` enum**. Every `raise SSRFBlockedError(..., reason="ssrf_...")` in `infrastructure/security/ssrf.py` and `infrastructure/security/blocklist.py` uses a string literal that must be a value of `SecurityBlockReason`. There is no runtime enforcement of this correspondence — it is a convention upheld by unit tests in `test_ssrf_validator.py` (which assert the specific `reason` string on each raised error). When adding a new rejection class to `ssrf.py`, add the reason string to `SecurityBlockReason` first, then use it in the raise site.

**PII detection runs on chunk text after tokenization but before vector embedding**. The Knowledge pipeline order is: raw text → `Tokenizer.split_sentences` → `Chunker.chunk` → `DefaultPIIDetector.detect_and_redact` → `EmbeddingProvider.embed_batch` → ChromaDB upsert. Embeddings are computed on the redacted text. For chunks that are heavy with PII (e.g., a document that is mostly contact information), the embedding quality may be degraded because `[REDACTED:EMAIL]` tokens carry no semantic weight. This is an acceptable trade-off: sending PII to an external embedding API (Gemini/OpenAI) is the alternative, which is worse. Ollama (local) is the one embedding backend that does not transmit data externally — it is the recommended backend for tenants with strict data-residency requirements.

**`sanitize_text` strips `<link>` and `<meta>` tags**. The `_DANGEROUS_TAGS` regex in `infrastructure/security/sanitize.py` includes `link` and `meta`. If a future report template relies on Markdown-to-HTML conversion that generates these tags for styling (e.g., a `<link rel="stylesheet">` in the output HTML), the sanitizer will strip them. The Publisher's HTML rendering path must inject stylesheets via a controlled template wrapper rather than generating them from analysis text. This is already the case as of Phase 9's Publisher implementation; do not change the template to embed stylesheet links in analysis-derived HTML.
