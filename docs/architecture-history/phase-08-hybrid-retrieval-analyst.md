# Phase 8 — Hybrid Retrieval + Analyst Agent (Full RAG Pipeline)

**Status**: ✅ Complete

## Summary

`shared/config` — `AnalysisSettings` controls the retrieval funnel: `retrieval_k=50` (per backend per sub-query) → `rerank_k=30` (after RRF) → `final_k=10` (after rerank). Adds `query_expansion_count=3`, `validator_threshold=0.75`, `max_refinement_loops=2`. Cohere keys already lived in `LLMSettings`.
`domain/retrieval/` — new module. `models.py`: `RetrievalRequest` (frozen, query/tenant_id/category_id/top_k), `RetrievedChunk` (chunk_id, score, text, parent_text, document_id, source_id, source_url). Score is unbounded — BM25/RRF/reranker have different ranges; only relative order matters. `protocols.py`: `@runtime_checkable` Protocols — `Retriever`, `BM25Retriever`, `VectorRetriever` (same shape, separate types for correct wiring in `HybridRetriever`), and `CrossEncoderReranker` (different signature — takes a candidate list, not a request).
`services/retrieval/rrf.py` — Reciprocal Rank Fusion as a pure function: `rrf(ranked_lists, *, k=60) → dict[str, float]`. Rank-only, no score normalisation. Empty lists return `{}`. k<1 raises. Comprehensive golden-example tests pin the formula.
`infrastructure/retrieval/mongo_bm25.py` — `MongoBM25Retriever` satisfies `BM25Retriever` structurally. Uses `TenantScopedRepository._tenant_query()` with `$text` + `category_id` filter, projects `{"$meta": "textScore"}` via the `chunks__text_bm25` index, sorts by it, limits by `top_k`. `parent_text`/`source_url` left None — filled by `HybridRetriever` / Analyst respectively. OTel span `"retrieval.bm25"`.
`infrastructure/retrieval/chroma_vector.py` — `ChromaVectorRetriever`: embeds the query with `task_type="query"`, builds the collection name via the same formula as `KnowledgePipeline` (`chunks__{tenant_short}__{model_id_safe}`), queries with Chroma `$and` filter (`tenant_id` + `category_id`). Reads text from `VectorHit.document`; skips hits with no text (logs warning). OTel span `"retrieval.vector"`.
`domain/llm/providers.py` — `EmbeddingProvider.embed_batch` extended with `task_type: Literal["document", "query"] = "document"` (Gemini honors via `RETRIEVAL_DOCUMENT`/`RETRIEVAL_QUERY`; OpenAI/Ollama accept and ignore). `RetryingEmbeddingProvider` passes through. New `StructuredChatProvider` Protocol with `complete_with_tools(messages, tools, *, tool_choice) → (name, args)` — Anthropic/Gemini/OpenAI chat providers all satisfy it structurally.
`infrastructure/retrieval/cohere_reranker.py` — `CohereReranker` (rerank-v3.5). Lifecycle: `connect()` creates `cohere.AsyncClientV2`, `close()` releases, `health_check()` returns client-set state (no free ping endpoint). `rerank(query, candidates, *, top_k)` replaces each chunk's score with `relevance_score`, preserves all other fields, clamps `top_n ≤ len(candidates)`. Error mapping: `TooManyRequestsError → LLMRateLimitError`, `Internal/Connection → LLMTransientError`, auth → `LLMPermanentError`. Empty input short-circuits without API call.
`services/retrieval/query_expansion.py` — `QueryExpander.expand(query, *, count=3)`. Loads prompt from `services/analysis/prompts/query_expansion.md` via `importlib.resources` — never inline. Parses LLM output via `_parse_sub_queries` (strips numbering/bullet prefixes via regex, dedups case-insensitively). On any LLM failure or empty response, falls back to `[original_query]` — single-query recall is the floor.
`services/retrieval/hybrid.py` — `HybridRetriever` orchestrates the 5-stage pipeline: (1) query expansion, (2) `asyncio.gather` over `2 × N` backend tasks (BM25 + vector per sub-query, `return_exceptions=True` so partial backend failure does not kill the run), (3) RRF merge → top `rerank_k`, (4) optional reranker (catches `Exception` → logs warning and falls back to RRF-sorted top-K), (5) parent expansion via two `MongoChunkRepository.get_many` batch calls. Spans on every stage.
`infrastructure/llm/anthropic.py` — real `AnthropicChatProvider` (was stub). `complete()` extracts system messages and applies `cache_control: {"type": "ephemeral"}` for prompt caching by default. `stream()` async-generates from `messages.stream()`. `complete_with_tools()` forces tool_choice, raises `LLMSchemaError` if no `tool_use` block. Finish-reason map: `end_turn→stop`, `max_tokens→length`, `tool_use→tool_calls`.
`infrastructure/llm/gemini.py` — real `GeminiChatProvider` alongside the existing embedding provider. Sync SDK wrapped in `asyncio.to_thread`. Translates `"assistant"` → `"model"` for the Gemini API. Tool definitions converted from Anthropic-style to `FunctionDeclaration` + `ToolConfig(mode=ANY)` internally. No prompt caching (Gemini context-cache is a separate billable feature with different semantics — intentionally not exposed). `types.Part.from_text(text=...)` is keyword-only in this SDK version.
`infrastructure/llm/openai.py` — real `OpenAIChatProvider` alongside embedding. Native `AsyncOpenAI` — no `asyncio.to_thread` needed. System messages stay in the messages array (unlike Anthropic/Gemini which extract). Tool args returned as JSON string → parsed via `json.loads`; bad JSON raises `LLMSchemaError`. `_map_openai_error` extracted as module-level helper; the embedding provider's catch refactored to use it.
`infrastructure/llm/registry.py` — added `build_chat_provider(settings)`. No retry wrapper — chat retry is the caller's responsibility (multi-step Analyst flow needs per-call control).
`services/analysis/prompts/` — three markdown files loaded via `importlib.resources`. `analyst_system.md`: persona + 7 hard rules (never invent chunk_ids, every claim cited, etc.) + `record_analysis` tool contract. `validator_system.md`: 5-dimension weighted rubric (grounding 0.30, faithfulness 0.25, quality 0.20, coverage 0.15, specificity 0.10) + explicit aggregation formula + `record_validation` tool contract. `query_expansion.md`: terse "one sub-query per line, no formatting" instruction. Smoke tests pin file existence and required tool/dimension keywords.
`services/analysis/validator.py` — `AnalysisValidator.validate(analysis, chunks, category) → ValidationResult`. Tool name pinned: `"record_validation"`. `ValidationResult` (frozen): `score ∈ [0,1]`, `passed: bool` (LLM's own judgment), `notes`, plus `meets_threshold(threshold)` helper. **Two pass signals**: the LLM's `passed` AND `score >= settings.validator_threshold` (orchestrator uses the latter as canonical). `_parse_validation_args` defensively coerces int score → float, clamps to [0,1], raises `LLMSchemaError` only on missing/non-numeric score.
`services/analysis/analyst.py` — `Analyst.produce_analysis(category, *, tenant_id, job_id, query=None, refinement_notes=None, refinement_loop=0) → AnalysisResult`. Tool name pinned: `"record_analysis"`. Five stages: query composition (appends refinement_notes if present), retrieval, structured LLM call via tool_use, validation, validator-field stamping via `model_copy` (Analysis has `extra="forbid"`). `_build_analysis_from_tool_args` is a defensive pure function: drops `supporting_chunk_ids` not in the retrieved corpus (insights with zero valid IDs after filtering are dropped entirely), dedups citations by `chunk_id`, truncates excerpts to 500 chars. Token accounting deferred (`complete_with_tools` doesn't return usage yet — Phase 9+ polish).
`orchestration/events.py` — `AnalysisPendingEvent` carries `(job_id, category_id, refinement_loop, refinement_notes, attempt)` — refinement loops are distinct work items, NOT collapsed by idempotency. `AnalysisCompletedEvent` carries `(job_id, success, analysis_id, passed, score, refinement_loop, error_code, error_message)`. `validation_score` renamed to `score` (no consumers existed). Three terminal states: `(success=True, passed=True)` happy, `(success=True, passed=False)` failed-validation (orchestrator decides refine vs graceful-degrade), `(success=False, analysis_id=None)` catastrophic.
`orchestration/workers/analyst_worker.py` — `AnalystWorker(BaseConsumer)`. Idempotency key: `f"analyst:{event.job_id}:{event.refinement_loop}"` — different loops are different work items. `handle()`: load Category (NotFoundError → permanent fail), call `analyst.produce_analysis`, persist Analysis to Mongo BEFORE publishing the completion event (orchestrator must be able to load by `analysis_id`). Two-tier failure: permanent (`LLMPermanentError`/`LLMSchemaError`/`ValidationError`/`NotFoundError`) → publish `success=False` + error code + analysis_id=None; transient → re-raise so `BaseConsumer` routes to `RETRY_ANALYSIS_30S → 5M → 1H → DLQ`. OTel span `"analyst.run_pass"`.
`orchestration/workers/orchestrator_worker.py` — now subscribes to FOUR topics: `JOBS_REQUESTED`, `INGEST_COMPLETED`, `KNOWLEDGE_COMPLETED`, `ANALYSIS_COMPLETED`. `_handle_knowledge_completed` now uses `interrupt_after=[NODE_ANALYZE]`. New `_handle_analysis_completed`: three branches — (a) `success=False` → fail the job; (b) `passed OR refinement_loop >= max_refinement_loops` → `aupdate_state(stage=PUBLISHING, analysis=...)` as_node=NODE_ANALYZE, stream forward through publish_node (graceful degradation: low-confidence result publishes if budget exhausted); (c) refine → load Analysis from Mongo to extract `validator_notes`, bump refinement_loops, publish new AnalysisPendingEvent (NOT via refine_node — orchestrator owns the republish, graph stays paused). Constructor takes `MongoAnalysisRepository` and `AnalysisSettings`.
`agents/orchestrator/nodes.py` — `retrieve_node` and `analyze_node` are now dual-mode (same pattern as ingest/embed). `retrieve_node` production: thin pass-through, populates `retrieved_chunk_ids` from `state.knowledge.chunk_refs` so `after_retrieve` routes forward (analyst worker ignores them and runs its own HybridRetriever); query set to `"delegated_to_analyst_worker"` marker. `analyze_node` production: publishes `AnalysisPendingEvent(refinement_loop=state.refinement_loops)` with OTel `traceparent` injected, returns `{"attempts": ...}` only (no AnalysisState, no transition). Stub paths preserved for unit tests. `refine_node` retained for the stub graph path; production bypasses it.
`docker/analyst.Dockerfile` — multi-stage build, dep groups `llm + rag + blob` (no `agents` — no LangGraph in the analyst worker). `docker-compose.app.yml` — adds `analyst-worker` service depending on mongo+kafka, with `LLM__DEFAULT_CHAT_PROVIDER=anthropic`, `LLM__DEFAULT_EMBEDDING_PROVIDER=gemini`, and API keys passed via `${VAR:-}` from `.env.local`. `Makefile up-app` build list now includes `knowledge-worker` and `analyst-worker`.
